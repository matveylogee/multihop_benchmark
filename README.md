# Multihop Benchmark

Репозиторий исследовательского проекта по созданию русскоязычного бенчмарка сложных multi-hop и multi-criteria запросов для оценки больших языковых моделей.

Основная цель проекта — собрать набор вопросов, на которых модель должна не просто вспомнить один факт, а совместить несколько условий, пройти по цепочке связанных сущностей, корректно обработать список подходящих ответов и не галлюцинировать там, где правильного ответа не существует.

Бенчмарк ориентирован на проверку следующих способностей LLM:

1. **Многошаговое рассуждение**  
   Модель должна соединять несколько фактов, а не отвечать на вопрос по одному прямому утверждению.

2. **Работа с несколькими критериями одновременно**  
   Ответ должен удовлетворять всем ограничениям из запроса: домену, жанру, стране, периоду времени, автору, участнику, рейтингу, свойствам объекта и другим фильтрам.

3. **Устойчивость к галлюцинациям**  
   В датасете есть запросы, для которых корректный ответ — пустое множество. Такие примеры проверяют, умеет ли модель не придумывать правдоподобные, но неверные ответы.

Основная часть бенчмарка строится на Wikidata: запросы генерируются по формальным шаблонам, а правильные ответы собираются через SPARQL. Также в проекте анализируются внешние QA-датасеты, из которых отбираются сложные примеры и полезные reasoning-паттерны.

## Мотивация

Современные языковые модели часто хорошо отвечают на простые фактологические вопросы, особенно если ответ часто встречался в обучающих данных или легко находится в поиске. Но этого недостаточно для оценки реальных поисковых и reasoning-систем.

В этом проекте проверяются более сложные случаи, где модель должна:

- удерживать несколько условий одновременно;
- объединять факты о разных сущностях;
- возвращать не один объект, а список всех подходящих ответов;
- не заменять исходное условие на похожее, но более простое;
- отличать правильные ответы от near-miss кандидатов;
- понимать, когда подходящих ответов нет;
- работать с прямыми и обратными формулировками фактов;
- опираться на retrieval/grounding, а не только на знания из pretrain.

Отдельная мотивация проекта связана с эффектом **Reversal Curse**: модель может знать факт в одном направлении, но ошибаться при обратной формулировке. В этом бенчмарке идея расширяется с простых одношаговых фактов на более сложные multi-hop и multi-criteria запросы.

## Основная идея

Бенчмарк состоит из русскоязычных запросов, для которых можно формально проверить правильные ответы.

Типичный пример запроса:

```text
Назови 3 фильма в жанре «криминал» с участием актера Леонардо ДиКаприо,
которые идут не более 3 часов и имеют рейтинг по IMDb от 7.5 и выше.
Перечисли все подходящие.
```

Для такого запроса в датасете хранятся:

- текст запроса на русском языке;
- домен;
- уровень сложности;
- структурированное описание ограничений;
- идентификаторы правильных ответов;
- русские названия правильных ответов;
- SPARQL-запрос или другой формальный источник gold-ответов;
- дополнительная метаинформация для валидации и анализа.

Главный принцип: правильные ответы должны быть воспроизводимы из формального источника. Для основного слоя таким источником является Wikidata.

## Слои датасета

Проект можно разделить на несколько логических слоев.

### 1. Основной слой Wikidata

Это главная часть бенчмарка.

В ней генерируются русскоязычные multi-hop и multi-criteria запросы на основе сущностей, свойств и ограничений из Wikidata. Для каждого запроса строится формальный SPARQL-запрос, с помощью которого собираются или валидируются правильные ответы.

Примеры доменов:

- кино;
- книги;
- картины;
- музыкальные альбомы;
- видеоигры;
- блюда;
- страны и исторические государства;
- университеты;
- аэропорты;
- программное обеспечение;
- смартфоны;
- автомобили;
- география;
- другие домены с большим количеством Wikidata-сущностей.

Основной Wikidata-слой проектируется так, чтобы он был:

- **формально заземленным** — ответы связаны с Wikidata ID;
- **проверяемым** — хранятся ограничения и SPARQL-запросы;
- **ориентированным на русский язык** — предпочтение отдается русским labels;
- **разбитым по сложности** — запросы распределены по уровням `L1`–`L5`;
- **списочным** — многие запросы требуют несколько правильных ответов, а не один.

Основные файлы слоя находятся здесь:

```text
wikidata/out_wikidata_benchmark/
```

Ключевые артефакты:

```text
dataset_main.jsonl
dataset_main.full_gold_enriched.jsonl
dataset_main.full_gold_enriched.diagnostics.csv
zero_gold.json
```

### 2. Внешний слой сложных примеров

Внешние датасеты используются не как готовый финальный бенчмарк, а как источник сложных паттернов рассуждения.

Главный внешний источник в репозитории — **KQA Pro**. Он полезен тем, что содержит compositional logical forms и сложные структуры вопросов:

- многошаговые цепочки;
- пересечения множеств;
- qualifier constraints;
- временные ограничения;
- сравнения;
- superlative reasoning;
- логические операции над сущностями.

Пайплайн KQA Pro включает:

1. импорт исходных данных;
2. фильтрацию примеров, пригодных для реконструкции;
3. подготовку выбранных вопросов к ручному переводу;
4. конвертацию отобранных и переведенных примеров в JSONL-формат.

Основные ноутбуки:

```text
kqapro/03_import_kqapro_fixed.ipynb
kqapro/04_filter_kqapro_for_reconstruction.ipynb
kqapro/05_prepare_kqapro_selected_for_translation_pipeline_with_json.ipynb
kqapro/06_convert_kqapro_csv_to_manual_ru_jsonl.ipynb
```

### 4. Zero-gold запросы для проверки галлюцинаций

Отдельный слой датасета — запросы, для которых правильный ответ отсутствует.

Такие примеры нужны, чтобы проверять, умеет ли модель честно сказать, что подходящих объектов нет, вместо того чтобы придумывать plausible-looking ответы.

Zero-gold запросы могут строиться на:

- невозможных временных ограничениях;
- несовместимых категориях;
- пустых пересечениях;
- противоречивых условиях;
- невозможных комбинациях доменных критериев.

Пример ожидаемого поведения:

```text
Запрос:
Назови фильмы, которые одновременно были выпущены до появления кинематографа
и получили современную кинопремию.

Ожидаемое поведение:
Модель должна ответить, что подходящих фильмов нет, потому что условия несовместимы.
```

## Формат данных

Основной формат хранения — **JSONL**: один JSON-объект на строку.

Пример записи:

```json
{
  "id": "cinema_l5_length_0168",
  "domain": "cinema",
  "complexity": "L5",
  "query_text_ru": "Назови 3 фильма в жанре «криминал» с участием актера Леонардо ДиКаприо, которые идут не более 3 часов и имеют рейтинг по IMDb от 7.5 и выше. Перечисли все подходящие.",
  "constraints": {
    "type": "cinema_l5_length",
    "genre_ru": "криминал",
    "actor_ru": "Леонардо ДиКаприо",
    "movie_length_range": [1, 180],
    "rating": ["imdb", ">=", 7.5],
    "source_dataset": "wikidata"
  },
  "requested_count": 3,
  "gold_answer_qids": ["kp:5437094", "kp:462682", "kp:81314"],
  "gold_answer_labels_ru": ["..."],
  "sparql_query": "...",
  "created_at": "2026-...",
  "is_advanced": true,
  "template_id": "cinema_l5_length",
  "template_family": "cinema",
  "gold_truncated": false,
  "ask_validator_sparql": "..."
}
```

### Основные поля

| Поле | Значение |
|---|---|
| `id` | Уникальный идентификатор запроса. Обычно содержит домен и уровень сложности. |
| `domain` | Домен запроса: например, `cinema`, `books`, `software`, `universities`. |
| `complexity` | Уровень сложности: обычно `L1`–`L5`. |
| `query_text_ru` | Текст запроса на русском языке. |
| `constraints` | Структурированное описание условий, из которых был собран запрос. |
| `requested_count` | Сколько ответов запрашивается в тексте вопроса. |
| `gold_answer_qids` | Идентификаторы правильных ответов. Для Wikidata это обычно QID, но для некоторых источников возможны source-specific ID. |
| `gold_answer_labels_ru` | Русские названия правильных ответов. |
| `sparql_query` | SPARQL-запрос, которым были получены gold-ответы. |
| `created_at` | Время создания записи. |
| `is_advanced` | Флаг сложного/расширенного шаблона. |
| `template_id` | ID конкретного шаблона генерации. |
| `template_family` | Семейство шаблонов. |
| `gold_truncated` | Флаг, показывающий, был ли gold-set обрезан лимитом. |
| `ask_validator_sparql` | Опциональный ASK-запрос для проверки кандидатного ответа. |

## Уровни сложности

В проекте используется пять примерных уровней сложности: `L1`–`L5`.

Точное значение уровня зависит от домена, но общая логика такая:

| Уровень | Описание |
|---|---|
| `L1` | Простые multi-criteria запросы. Обычно прямые фильтры: жанр + год, страна + тип, категория + период. |
| `L2` | Немного более сложные пересечения условий, часто с дополнительным фильтром или менее прямым свойством. |
| `L3` | Более сильная композиционность: дополнительные ограничения, рейтинги, qualifiers, свойства связанных сущностей. |
| `L4` | Продвинутые multi-hop запросы: bridge-сущности, исключения, вложенные ограничения, нетривиальная логическая структура. |
| `L5` | Самые сложные запросы: несколько цепочек условий, негативные ограничения, рейтинг/длина/дата, source-specific свойства, трудный контроль размера gold-set. |

Эти уровни не претендуют на идеально строгую теоретическую иерархию. Их задача — дать удобные срезы для анализа качества моделей.

## Общий пайплайн генерации и оценки

Общий workflow проекта:

```text
1. Сгенерировать или импортировать кандидатные запросы
2. Нормализовать записи в общий JSONL-формат
3. Отфильтровать слишком простые, шумные или некорректные примеры
4. Собрать или провалидировать gold-ответы
5. Обогатить gold-set при необходимости
6. Подготовить русские формулировки, перевод или перефразирование
7. Конвертировать JSONL в downstream-формат для инференса
8. Запустить модели на запросах
9. Оценить ответы через автоматические проверки и LLM-as-a-Judge
10. Проанализировать качество по доменам, сложности и типам запросов
```

Репозиторий в основном содержит публичную/offline-часть этого процесса:

- ноутбуки генерации через Wikidata;
- ноутбуки импорта и фильтрации внешних датасетов;
- промежуточные артефакты;
- JSONL-файлы;
- diagnostics и manual review файлы.

Некоторые downstream-этапы инференса и оценки могут зависеть от внутренней инфраструктуры.

## Пайплайн генерации через Wikidata

Wikidata — основной источник данных для бенчмарка.

Главный ноутбук:

```text
wikidata/multihop_benchmark.ipynb
```

Он отвечает за генерацию запросов и сбор gold-ответов.

Типичный этап генерации:

1. выбирается домен;
2. сэмплируются сущности, свойства и ограничения;
3. строится русскоязычный текст запроса;
4. строится SPARQL-запрос;
5. SPARQL-запрос выполняется к Wikidata;
6. невалидные примеры отбрасываются;
7. сохраняются gold QID и labels;
8. запись добавляется в JSONL.

В пайплайне контролируются следующие проблемы:

- пустые gold-set, если запрос не является zero-gold;
- слишком большие answer-set;
- отсутствие русских labels;
- дубликаты;
- слишком простые запросы;
- некорректные комбинации условий;
- сохранение формального источника ответа;
- воспроизводимость сгенерированных примеров.

Дополнительное обогащение gold-ответов выполняется ноутбуком:

```text
wikidata/enrich_existing_wikidata_gold.ipynb
```

Он нужен, когда уже существующие записи требуется переобработать или дополнить более полным списком gold-ответов без потери исходной структуры.

## Пайплайн внешних датасетов

Внешние датасеты используются иначе, чем Wikidata.

Они не считаются одинаково чистым формальным источником правды. Скорее, они помогают найти дополнительные сложные reasoning-паттерны и вручную отобрать хорошие примеры.

Типичный workflow для внешнего датасета:

```text
1. Импортировать исходный датасет
2. Привести записи к промежуточному формату
3. Отфильтровать примеры по сложности и пригодности
4. Подготовить таблицу для ручной проверки
5. Перевести или переписать выбранные вопросы на русский
6. Конвертировать проверенные записи в общий JSONL-формат
7. При необходимости объединить с основным benchmark-форматом
```

## Пайплайн оценки моделей

Бенчмарк можно использовать для оценки обычных LLM и retrieval/search-augmented систем.

Полный evaluation workflow может включать:

1. **Подготовку входов**  
   Конвертацию JSONL-записей в формат, нужный для инференса.

2. **Инференс модели**  
   Запуск модели на русскоязычных запросах.

3. **Нормализацию ответа**  
   Извлечение сущностей или строковых ответов из вывода модели.

4. **Сопоставление с gold**  
   Сравнение предсказанных сущностей или labels с правильными ответами.

5. **LLM-as-a-Judge**  
   Оценку ответов judge-моделью, если exact matching недостаточен.

6. **Анализ по срезам**  
   Агрегацию метрик по:
   - доменам;
   - уровням сложности;
   - template family;
   - zero-gold / non-zero-gold;
   - прямым и обратным формулировкам;
   - количеству запрошенных ответов;
   - типам логических ограничений.

В более широком workflow проекта JSONL-записи могут конвертироваться в таблицы, обрабатываться через YQL, запускаться через inference pipeline и оцениваться с помощью автоматических проверок и LLM-as-a-Judge.

Этот репозиторий в первую очередь отвечает за подготовку и воспроизводимое хранение benchmark-данных.

## Как пользоваться репозиторием

### 1. Клонировать репозиторий

```bash
git clone https://github.com/matveylogee/multihop_benchmark.git
cd multihop_benchmark
```

### 2. Открыть нужные ноутбуки

Для основного Wikidata-слоя:

```text
wikidata/multihop_benchmark.ipynb
wikidata/enrich_existing_wikidata_gold.ipynb
```

Для KQA Pro:

```text
kqapro/03_import_kqapro_fixed.ipynb
kqapro/04_filter_kqapro_for_reconstruction.ipynb
kqapro/05_prepare_kqapro_selected_for_translation_pipeline_with_json.ipynb
kqapro/06_convert_kqapro_csv_to_manual_ru_jsonl.ipynb
```

Для Mintaka:

```text
mintaka/03c_import_mintaka_multihop.ipynb
mintaka/04c_filter_mintaka_for_reconstruction.ipynb
mintaka/05c_convert_mintaka_csv_to_manual_ru_jsonl.ipynb
```

Для 2WikiMultiHopQA:

```text
2wikimultihop/03e_import_2wikimultihop.ipynb
2wikimultihop/04e_filter_2wikimultihop_for_reconstruction_fixed_v5.ipynb
```

### 3. Посмотреть сгенерированные артефакты

Главные выходные файлы лежат в:

```text
wikidata/out_wikidata_benchmark/
```

Самые важные файлы:

```text
dataset_main.jsonl
dataset_main.full_gold_enriched.jsonl
dataset_main.full_gold_enriched.diagnostics.csv
zero_gold.json
```

### 4. Провалидировать данные перед финальной оценкой

Перед использованием датасета для финальной оценки желательно проверить:

- что все обязательные поля заполнены;
- что у gold-ответов есть русские labels;
- что gold-set не был случайно обрезан;
- что `requested_count` согласован с количеством gold-ответов;
- что zero-gold запросы действительно не имеют валидных ответов;
- что внешние примеры прошли ручную проверку.

## Текущий статус

Репозиторий находится в активной исследовательской разработке.

Основные направления работы:

- расширение Wikidata-ядра;
- улучшение покрытия по доменам;
- увеличение количества сложных `L4`–`L5` запросов;
- удаление слишком простых и шумных примеров;
- обогащение gold-set;
- конвертация внешних датасетов в единый формат;
- подготовка данных для инференса и LLM-as-a-Judge оценки.

На текущий момент в репозитории есть:

- основной ноутбук генерации через Wikidata;
- Wikidata JSONL-артефакты;
- enriched gold файлы;
- diagnostics;
- KQA Pro import/filter/translation pipeline;
- RuBQ/MKQA import pipeline;
- Mintaka import/filter/conversion pipeline;
- 2WikiMultiHopQA import/filter pipeline;
- manual review и промежуточные artifacts.

## Ограничения

Это исследовательский benchmark-building репозиторий, а не финальный leaderboard-пакет.

Важные ограничения:

- покрытие Wikidata неравномерно по доменам;
- некоторые домены sparse или noisy;
- часть сгенерированных записей требует ручной валидации;
- русские labels доступны не для всех сущностей;
- сложные SPARQL-запросы могут выполняться медленно;
- во внешних датасетах много слишком простых примеров;
- exact string matching часто недостаточен для оценки ответов модели;
- LLM-as-a-Judge полезен, но не является идеальным источником истины.

Поэтому в датасете сохраняется максимум audit-информации:

- constraints;
- SPARQL-запросы;
- gold identifiers;
- gold labels;
- template metadata;
- diagnostics.

## Назначение проекта

Итоговая цель проекта — построить надежный русскоязычный бенчмарк для оценки того, насколько хорошо LLM и search-augmented LLM системы справляются со сложными grounded queries.

Бенчмарк особенно полезен для проверки, умеет ли модель:

- корректно совмещать несколько условий;
- извлекать и объединять факты о разных сущностях;
- отвечать на list-style запросы;
- работать с обратными и непрямыми отношениями;
- не галлюцинировать в случаях с пустым ответом;
- объяснять отсутствие ответа при противоречивых условиях;
- стабильно работать на разных доменах и уровнях сложности.

Репозиторий одновременно является рабочим пространством для построения датасета и воспроизводимым описанием процесса его создания.
