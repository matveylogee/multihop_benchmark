# Multihop Benchmark

Русскоязычный исследовательский бенчмарк для оценки способности LLM отвечать на сложные multi-hop и многокритериальные запросы.

Проект посвящён проверке того, насколько языковые модели умеют:
- удерживать несколько условий в одном запросе;
- объединять факты из разных источников;
- корректно работать с list-style ответами;
- не подменять ограничения похожими, но неверными;
- не галлюцинировать ответы, если корректных gold-ответов не существует.

## Основная идея

Бенчмарк строится вокруг русскоязычных запросов, где ответ нельзя получить простым извлечением одного факта. Многие вопросы требуют пересечения нескольких условий: домен, страна, жанр, период, автор, участник, география, награды, свойства объекта и другие ограничения.

Основной слой данных собирается через Wikidata: для запросов автоматически строятся constraints и проверяются gold-ответы. Дополнительно рассматриваются внешние multi-hop / complex QA датасеты как источники сложных reasoning-паттернов.

## Структура репозитория

```text
multihop_benchmark/
├── wikidata/          # основной пайплайн генерации запросов и gold-ответов через Wikidata
├── kqapro/            # импорт, фильтрация и подготовка сложных паттернов из KQA Pro
├── rubq_mkqa/         # анализ RuBQ / MKQA как источников русских формулировок и KBQA-шаблонов
├── mintaka/           # анализ Mintaka как дополнительного источника multi-hop кандидатов
└── 2wikimultihop/     # анализ 2WikiMultiHopQA как источника comparison / bridge-comparison задач
```

## Слои датасета

Итоговая логика бенчмарка предполагает несколько слоёв:

### 1. Wikidata core

Основной слой датасета. Включает русскоязычные multi-hop и многокритериальные запросы с формально проверяемыми gold-ответами.

Для каждого примера могут храниться:
- текст запроса на русском;
- домен;
- уровень сложности;
- constraints;
- gold answer labels;
- gold answer QIDs;
- SPARQL или структурированное описание запроса.

### 2. External complex layer

Дополнительный слой сложных запросов и reasoning-паттернов из внешних датасетов.

Основной внешний источник — KQA Pro. Он используется не как готовый русский датасет, а как донор сложных логических схем: multi-hop, logical set operations, qualifier constraints, temporal conditions, comparison / superlative reasoning.

### 3. Auxiliary external splits

Некоторые внешние источники оказались полезны только частично:

- RuBQ / MKQA — полезны как источники русскоязычных формулировок и простых KBQA-шаблонов;
- Mintaka — даёт отдельные multi-hop кандидаты, но содержит много shallow bridge-вопросов;
- 2WikiMultiHopQA — полезен как источник comparison-heavy и bridge-comparison задач, но не как основной слой benchmark-core.

### 4. Zero-gold / hallucination probes

Отдельный слой запросов, для которых корректных ответов быть не должно.

Такие запросы используются для проверки того, умеет ли модель:
- не выдумывать ответы;
- явно говорить, что подходящих объектов нет;
- объяснять, почему условия противоречивы или дают пустое множество.

Примеры типов zero-gold запросов:
- `temporal_impossibility`;
- `historical_impossibility`;
- `mutually_exclusive_filters`;
- `empty_intersection`;
- `category_contradiction`;
- `hard_contradiction`.

## Пайплайн

Общий пайплайн работы с данными:

1. Генерация или импорт кандидатов.
2. Нормализация в единую схему.
3. Фильтрация слишком простых или нерелевантных примеров.
4. Проверка gold-ответов.
5. Подготовка данных к переводу / рефразированию.
6. Загрузка в инфраструктуру для инференса моделей.
7. Оценка ответов моделей, в том числе через LLM-as-a-Judge.

## Используемые источники

В проекте рассматриваются следующие источники:

- **Wikidata** — основной источник истины и gold-ответов.
- **KQA Pro** — основной внешний донор сложных compositional patterns.
- **RuBQ / MKQA** — вспомогательные источники русских формулировок и KBQA-шаблонов.
- **Mintaka** — дополнительный источник отдельных multi-hop примеров.
- **2WikiMultiHopQA** — дополнительный источник comparison / bridge-comparison задач.

## Назначение проекта

Бенчмарк предназначен для исследовательской оценки LLM и search-augmented LLM-систем на задачах, где важно не просто знание отдельного факта, а способность корректно собрать ответ по нескольким ограничениям.

Особый интерес представляют случаи, где модель:
- должна найти несколько подходящих объектов;
- должна учитывать все ограничения сразу;
- должна не галлюцинировать при пустом множестве ответов;
- должна отличать похожие, но некорректные ответы от настоящих gold-ответов.

## Статус

Проект находится в исследовательской разработке. Ноутбуки в репозитории отражают этапы анализа источников, отбора кандидатов и подготовки данных для дальнейшего benchmark-пайплайна.
